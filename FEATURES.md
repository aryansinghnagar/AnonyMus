# System Features & Cryptographic Specification (P2P Branch)

This document details the functional features and cryptographic specifications implemented in the decentralized P2P version of the AnonyMus application.

---

## 1. Cryptographic Security Model

The decentralized P2P architecture implements a fully peer-to-peer end-to-end encrypted messaging protocol. There are no intermediate servers or key relays. All key exchanges and message decryptions are processed directly on client host machines.

### A. Diffie-Hellman Handshake
- **Parameters**: Node key generation is performed using **NIST P-256 (secp256r1)** elliptic curve cryptography.
- **Out-of-Band Invitation**: Initial public keys and onion addresses are exchanged out-of-band using secure contact invite links.
- **Shared Secret Derivation**: Once a handshake invitation is accepted, both nodes perform an Elliptic Curve Diffie-Hellman (ECDH) operation to derive a 256-bit shared master key, which is advanced using **HKDF-SHA256** to create outbound and inbound message keys.

### B. Local Database AES-GCM Encryption
To protect sensitive cryptographic secrets in the event of local device theft:
- **Key Derivation**: When logging into the local panel, the master password is run through **PBKDF2-HMAC-SHA256** with a static salt and 10,000 iterations to derive a local database key.
- **Symmetric Encryption**: Derived session keys and peer public keys are stored encrypted on disk in the local SQLite database (`local_node.db`) using **AES-256-GCM** with a random 12-byte initialization vector (IV) generated by a CSPRNG.
- **RAM Protection**: Plaintext credentials and database keys reside strictly in ephemeral session memory and are cleared upon user logout or daemon shutdown.

## 2. Tor Onion Routing Integration

To achieve complete anonymity and circumvent network firewalls, the application embeds a managed Tor client.

- **Embedded Tor Client**: The server dynamically downloads, extracts, and executes the **Tor Expert Bundle**.
- **Onion Hidden Service**: The node configures a local Tor Onion Service (`.onion`), exposing local Flask endpoint listeners securely to the Tor network. Remote peers communicate with this hidden service over port `80`, which Tor tunnels to a dynamically assigned local port.
- **Outbound SOCKS5 Routing**: Outbound messages and handshake requests to peer onion services are routed through Tor's local SOCKS5 proxy port (default: `9050`). This prevents any direct peer-to-peer IP connection, concealing the physical locations of both nodes.
- **Local Host Bound Security**: The local Flask server is bound strictly to loopback interfaces (`127.0.0.1`). Local control endpoints are rejected if accessed from any Host header other than `localhost` or `127.0.0.1`, preventing remote Tor users from accessing the node administration interface.

## 3. Disguised GUI Launcher

For security and concealment, the execution wizard is packaged behind a camouflage interface:
- **Network Adapter Camouflage**: The GUI presenting status controls is designed to look like a "Windows Network Diagnostics & Adapter Utility".
- **Dynamic Port Assignment**: In this mode, control panel web ports and Tor SOCKS ports are dynamically allocated based on system availability, evading static port detection by local monitors.
- **Camouflaged Configuration**: Service states, Tor bootstrapping indicators, and system spec diagnostics are rendered to resemble standard system network verification logs.

## 4. Secure Wiping Uninstaller

To ensure absolute clean removal, the Inno Setup uninstaller (`setup.iss` compilation) contains secure cleanup actions:
- **Database Erasure**: The uninstaller prompts the user to wipe all local data. If confirmed, it deletes the SQLite database files (`users.db`, `local_node.db`) and custom settings files.
- **Binary Cleanup**: It recursively deletes the embedded Tor Expert Bundle binaries (`bin/` directory) and all temporary files, leaving no trace of the application on the host filesystem.
