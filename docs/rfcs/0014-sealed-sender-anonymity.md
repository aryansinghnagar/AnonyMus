# RFC 0014: Sealed Sender Anonymity Envelope

- **Status:** Approved
- **Author(s):** AnonyMus Contributors
- **Created:** 2026-07-13
- **Updated:** 2026-07-13

---

## 1. Context

Intermediate relay servers route messages based on recipient queue IDs, but do not need to know who sent the message. This RFC defines the Sealed Sender envelope format which encrypts the sender's identity from the relay.

## 2. Goals & Non-Goals

### Goals
- Hide the sender's username, signing key, and DH key from the intermediate relay.
- Bind the payload to a symmetric Sealed Sender Key shared between communicating parties.

### Non-Goals
- Hide IP addresses from the relay (handled via Tor egress routing).

## 3. Design Details

The Sealed Sender envelope structure is:
```json
{
  "recipient_queue_id": "string",
  "encrypted_payload": "base64_encoded_bytes"
}
```
The `encrypted_payload` is generated by encrypting a `SealedSenderPayload` structure:
```json
{
  "sender_username": "string",
  "sender_signing_pub": [32],
  "sender_dh_pub": [32],
  "inner_message": "bytes"
}
```
Encryption is performed using AES-256-GCM under the `sealed_sender_key` shared during key agreement.

## 4. Security & Privacy Implications

- **Relay Blindness**: Relays see only a flat queue ID and an encrypted blob. They cannot construct a metadata social graph of who is communicating with whom.
- **AEAD Protection**: Protects payload integrity and authenticity.

## 5. Backward Compatibility

- Supported only in `protocolVersion >= 3`. Legacy clients will continue to expose plaintext sender headers to the relay during transitions.
